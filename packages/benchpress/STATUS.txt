OPEN — KNOWN ISSUES (in rough priority order)
---------------------------------------------

P0  Anthropic prompt cache is broken
    Per-call payload shows cache_creation_input_tokens = 16,988 — the
    cache_control marker is being set ~17K into the ~210K prompt instead
    of after the whole static prefix. Cache hit rate is 8% on Sonnet 4.6;
    GPT-5.5 hits 99.6% (OpenAI auto-caches prefixes server-side).
    Impact: 5× cost penalty for every Anthropic call in the matrix.
    Fix: probably one line in packages/llm-service/src/providers/anthropic
    .ts to move the cache_control breakpoint. Untouched so far.

P0  Prompt size is ~200K tokens per call
    This is the *real* cost driver. $1 for a "what was my weight"
    question is not sustainable; a full matrix today is ~$100. SCM
    composes 25+ modules and 7 background loaders fire context into
    every call — that's where the bulk lives. Mitigation discussion is
    its own thread (see DESIGN QUESTIONS below).

P1  Matrix runner exhibits "agent silence" under some conditions
    Observed twice:
      - claude-haiku-4-5 × q01/q03: ONE llm_invocation event with
        prompt_tokens=0, completion_tokens=0, output={thoughts:"",code:""},
        loop=4. StreamingRunnerV3 retried up to its limit of empty
        structured outputs.
      - claude-sonnet-4-6 × q06/q07 (post-embeddings, matrix runner):
        ZERO llm_invocation events. sendMessageAsync logs status=ok in
        simi_step, then nothing for 180s.
    The two patterns may not have the same root cause. Haiku looks like
    a model/runner schema-handshake issue ("doesn't produce valid
    structured output for streaming_v3_structured"). Sonnet's silent
    failure in the matrix runner — unclear; possibly a regression in
    run_bench.ts vs the simpler factory_smoke spec, or post-embedding
    side effect.

P1  Workflow timeout is too generous
    bench_answer waitFor is 180s. When the model hangs/silent-fails,
    every scenario costs 3 min wall before the timeout fires. A 2×2
    failure run was 6 min today. Should be ~30-60s + fail-fast on
    "no llm_invocation N seconds after sendMessageAsync".

P2  Seed reset between models not wired
    q08 (mutation: log 30 pages of Dune on 2026-06-08) accumulates +30
    pages per model run, so multi-model q08 reads a drifting baseline.
    Currently q08 is skipped by default in run_bench.ts. Needs an
    in-place purge (DELETE FROM logs WHERE ... + re-import via
    `surreal sql --file ...`) between models.

P2  prompt_tokens=0 in llm_invocation payload on empty-output calls
    Either the local server's emitter isn't reading the count or the
    response is failing before parse. Conflated with the streaming_v3
    "empty structured output" bug. Worth tracing but distinct from the
    bigger prompt-cost issue.

P3  Untested models
    claude-opus-4-7, claude-opus-4-5, gemini-3.1-pro-preview, gemini-3
    -flash-preview, gemini-3.1-flash-lite, gpt-5.2, gpt-5-nano not yet
    exercised. Should wait for Anthropic cache fix + agent-silence
    diagnosis before doing the cheap-but-slow first sweep.


DESIGN QUESTIONS BEING THOUGHT THROUGH
--------------------------------------

(1) SCM module composition as a benchmark dimension
    The 25+ module SCM is the dominant cost source AND probably affects
    correctness/latency/UX-directive adherence in non-obvious ways. Treat
    SCM composition as a configurable axis of the benchmark — sweep
    several SCM variants (full / lean / minimal) × models × scenarios
    and report the (correctness, $-per-turn, directive-adherence) cube.
    Open: what variants exactly? "full prod", "lean (drop ancillary
    modules)", "minimal (only data + KG + core)" feel like the obvious
    three starting points.

(2) Dynamic / on-demand module loading
    Agent boots with a small index ("I have X tools/modules available;
    call open_module(name) to load it"). When it needs a capability, it
    loads the module's full system_msg + functions. When done, it can
    close_module(name) to evict.
    Implications:
      - Dramatic cost reduction for simple queries (no need for KG,
        web_search, scripture, etc. for "what was my weight")
      - Adds reasoning overhead (model must know what it needs)
      - Could *help* weak models (less context to navigate) or *hurt*
        them (need metacognition)
      - Benchmark gets richer: now we measure "did the agent open the
        right modules?" alongside correctness
    Mechanism sketch:
      - SCM gains a "core" + "expandable" partition
      - At boot: only "core" + index of expandable modules included
      - open_module / close_module tools exposed in core
      - Module state tracked in workspace (or sandbox state)
      - Re-open is cheap (cached system_msg blocks)

(3) Cache-aware module ordering
    Even with the full SCM, putting the *static* modules (intro, values,
    platform, scoping, responsiveness, conversational, …) before the
    *dynamic* ones (todos_context, metrics_context, kg) lets prompt
    caching catch the static prefix. Cheap win independent of (1)/(2).
